# 项目梳理：电商搜广推多目标学习模型 (CTR & CVR)

## 一、 业务场景与数据概况

- **业务场景**：基于经典的 Ali-CCP（阿里巴巴点击与转化预测）公开数据集。这是一个典型的电商搜广推多目标学习场景。
- **数据集地址**：[Ali-CCP Dataset](https://tianchi.aliyun.com/dataset/408)
- **预测目标**：
  - **CTR (点击率)**：衡量用户是否对曝光商品感兴趣。
  - **CVR (转化率)**：衡量用户点击后是否会产生真实的购买行为。
- **数据痛点（极端稀疏与不平衡）**：数据集展现了极端的漏斗效应。整体的点击正样本（CTR）占比大约在 **4%** 左右，而购买正样本（CVR）占比仅有极端的 **0.03%**（万分之三左右）。这构成了前期模型梯度震荡、负迁移以及容易退化为“瞎猜”（AUC 0.50）的根本物理原因。
  - **证据**：
    - 训练集：`[Click/CTR 均值]: 0.0466` | `[Conversion/CVR 均值]: 0.000286`
    - 测试集：`[Click/CTR 均值]: 0.0469` | `[Conversion/CVR 均值]: 0.000349`

## 二、 数据存储与处理流程 (`Opendata.ipynb` & `dataset.py`)

### 1. 淘宝储存策略
淘宝的数据量太大了，为了防止文件爆炸，它把数据拆成了两张表，类似于关系型数据库的主外键：
- **样本骨架表 (Sample Skeleton)**：存着标签（点没点、买没买）、独有特征，以及一个关键的“接头暗号（外键 MD5）”。
- **公共特征表 (Common Features)**：存着那些被很多人共享的特征集合。

<div align="center">
  <img src="https://tianchi-public.oss-cn-hangzhou.aliyuncs.com/public/files/forum/159463342659031201594633426409.png" width="400" />
</div>


**样本骨架表结构解析：**
1. **样本 ID 部分**：唯一标识样本骨架中的一条记录，取值从 1 开始直到全部样本数，是样本骨架文件的主键。
2. **标签部分**：包含一次浏览记录上的两种类型的标签信息：点击和转化事件是否发生。
3. **特征部分**：包含三个域：
   - `common_feature_index` 域：作为样本骨架文件的外键，用来关联公共特征文件中的信息，复原样本时使用。
   - `feature_num` 域：指出本条记录中特征列表 (`feature_list`) 包含的特征总数，是一个非负整数。
   - `feature_list` 域：由 ASCII 字符 `0x01` 分割的若干特征组成。特征数据结构由 ASCII 字符 `0x02` 和 `0x03` 作为分隔符构成，例如：`feature_field_id 0x02 feature_id 0x03 feature_value`。

### 2. 文件合并策略 (SQLite 极速查询)
- **官方定义**：由于商品等特征在多条样本中重复出现，为了节省存储，官方将数据拆分成了 Skeleton(骨架) 和 Common(公共特征)，需要通过 `common_feature_index` 进行 LEFT JOIN 关联。
- **代码动作**：由于 Common 表过大（超过 8GB），直接用 Pandas Merge 会导致内存溢出 (OOM)。代码巧妙地编写了 `build_common_sqlite_db` 函数，将 Common 表存入本地 SQLite 数据库并建立主键索引。在处理 Skeleton 数据时，按 Batch 取出 `common_idx` 去数据库中极速查询并合并，完美实现了低内存消耗的数据复原。

### 3. 清理异常数据
清洗掉（点击=0, 转化=1）这种非法漏斗数据：
```python
chunk = chunk[~((chunk['click'] == 0) & (chunk['conversion'] == 1))]
```

## 三、 特征工程与分组
### 1. 特征分组定义
```Python
FIELD_GROUP = {
    'epnet_scene': ['301'], # 走下路进入 EPNet 做全局缩放
    'user_profile': ['101', '121', '122', '124', '125', '126', '127', '128', '129'], # 走上路提供给 Gate NN 与 PPNet
    'user_behavior': ['109_14', '110_14', '127_14', '150_14'], # 同上，走上路的个性化门控
    'item_and_cross': ['205', '206', '207', '210', '216', '508', '509', '702', '853'] # 走底座并被“CTR塔”吸收
}
```
### 2. 字段含义与形态转换
举个例子：假设对于某一条样本，它的 user_profile 包含：
- 性别(fid=121)：feat_id='男', feat_val=1.0

- 年龄段(fid=122)：feat_id='25-30', feat_val=1.0

代码处理完后，并不会保持多维度的层次结构，而是直接拼装成一个长列表（每一个特征组都包含 _idx 供 Embedding 查表使用，以及 _val 作为权重数值，后续模型中主要使用 _idx）：

- user_profile_idx = [词表映射后的'男'的ID, 词表映射后的'25-30'的ID]

- user_profile_val = [1.0, 1.0]

**核心输入特征域含义:**

- epnet_scene_idx：业务场景特征（映射自原始 301 域）。样例：[2]，说明这条数据处在被编码为 2 的业务场景下。

- user_profile_idx：用户的基础社会属性（如用户 ID、性别、年龄、消费水平等）。样例：[1,4,2,2,2,2,2,5]。

- user_behavior_idx：用户在类目、店铺、品牌上的历史行为累积（以 14 结尾的域）。样例：[117,1290,116...]，长度为 50 的序列。

- item_and_cross_idx：当前曝光商品的信息（商品 ID、类目等）以及商品与用户的交叉组合特征。

### 3. 数据整理 (DataLoader)
collate_fn_pad(batch)：配合 DataLoader 使用的数据整理补齐函数。它会一次性收到 N 行数据 (batch_size)，找出这批数据里的最长长度，然后自动 Padding 0。

## 五、 模型训练与核心工程细节

### 1. 损失函数与对抗极端不平衡 (Focal Loss)
彻底废弃了原先标准的 BCE Loss 以及容易导致梯度爆炸的“乘以500倍硬核惩罚”，全面换用何恺明提出的 Focal Loss：
- **CTR 任务**：`FocalLoss(alpha=0.75, gamma=2.0)`。正样本约 4%，给予中等偏上的平衡权重。
- **CVR 任务**：`FocalLoss(alpha=0.95, gamma=2.0)`。正样本仅万分之三，给予极端的关注度。
- **作用**：模型被迫把所有的注意力 (Focus) 全集中在那些容易判错的困难样本上，抹平 Loss 震荡。

### 2. 优化器与学习率调度 (Optimizer & Scheduler)
- **优化器 (AdamW)**：`optim.AdamW(model.parameters(), lr=5e-4, weight_decay=1e-5)`。相比普通的 Adam，AdamW 能更有效地防止模型参数在稀疏特征域上过度膨胀，起到 L2 正则化的作用，提升了泛化能力。
- **调度器 (Cosine Annealing)**：`CosineAnnealingLR(optimizer, T_max=5, eta_min=1e-5)`。学习率没有采用死板的固定值或阶梯下降，而是像余弦曲线一样，在 5 个 Epoch 内从 `5e-4` 极其丝滑地降落到微小的 `1e-5`。初期大步寻优，末期精细微雕。

### 3. 深度网络的防御机制 (Gradient Clipping)
- **工程实现**：在每次 `loss.backward()` 后插入安全锁：`torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=5.0)`。
- **核心作用**：**防止梯度爆炸 (Gradient Explosion)**。架构中存在大量的“门控结构”（如 EPNet、Bridge Gate、PPNetLayer）包含 Sigmoid 连乘操作。此代码强制缩小异常梯度，像发动机转速限制器一样确保训练永不崩盘。

### 4. 早停机制与巅峰护航 (Early Stopping)
- **机制**：每个 Epoch 结束时监控 `current_auc_sum = ctr_auc + cvr_auc`。
- **容忍度 (Patience)**：设为 `2`。
- **护航操作**：发现新高立刻执行 `torch.save(model.state_dict(), save_path)`。物理拦截过拟合，避免模型死记硬背稀疏正样本导致验证集 AUC 断崖式下跌。

## 六、 参数设定字典

### 1. 训练超参数 (Training Hyperparameters)
不改变模型结构，专门用于控制模型的学习节奏和寻优过程。

| 参数名 | 当前设定值 | 所在文件 | 作用与调整建议 |
| :--- | :--- | :--- | :--- |
| `BATCH_SIZE` | 2048 | `mainV6` | 单次喂入数据量。若遇局部最优，可降至 1024 增加梯度噪声。 |
| `LEARNING_RATE` | 5e-4 | `mainV6` | 初始学习率。配合 AdamW 优化器，决定参数更新步长。 |
| `EPOCHS` | 5 | `mainV6` | 最大训练轮数。受早停机制保护，通常不需要设得过大。 |
| `weight_decay` | 1e-5 | `mainV6` | AdamW 权重衰减系数(L2正则化)，防止 Embedding 权重无限膨胀。 |
| `eta_min` | 1e-5 | `mainV6` | 余弦退火调度器的最低学习率，确保训练末期极细粒度收敛。 |
| `Focal Loss alpha`| CTR:0.75 / CVR:0.95| `mainV6` | 类别平衡权重。CVR正样本极少，因此给予 0.95 的极高权重。 |
| `Focal Loss gamma`| CTR:2.0 / CVR:2.0 | `mainV6` | 难易样本聚焦因子。若 CVR 依然学不好，可将 CVR 调高至 3.0。 |
| `max_norm` | 5.0 | `mainV6` | 梯度裁剪阈值。防止深层门控网络发生梯度爆炸。 |
| `patience` | 2 | `mainV6` | 早停容忍度。连续 2 个 Epoch 总 AUC 未提升则强行中止训练。 |

### 2. 模型架构超参数 (Architecture Hyperparameters)
决定模型“脑容量”的大小。

| 参数名 | 当前设定值 | 所在文件 | 作用与调整建议 |
| :--- | :--- | :--- | :--- |
| `EMBED_DIM` | 32 | `mainV6` | 所有特征基础表达维度。算力允许下可升至 64 获取更强语义表征。 |
| `shared_bottom_units`| [256, 128] | `mainV6` | CTR 共享底座隐藏层大小。最终输出 128 维特征供上方网络使用。 |
| `cvr_tower_units` | [128, 64] | `mainV6` | CVR 转化塔隐藏层大小。呈漏斗状递减，最后一层映射为 1 维标量。 |
| `BOTTOM_DROPOUT` | 0.0 | `mainV6` | CTR 底座随机失活率。目前未开启。 |
| `CVR_DROPOUT` | 0.05 | `mainV6` | CVR 塔随机失活率。极小的 Dropout 防止死记硬背稀疏样本。 |
| `attention_factor` | 16 | `modelV5`| AFM 交叉层内部注意力网络大小。16 维足以衡量特征组合重要性。 |
| `hidden_units (DIN)` | [64, 32] | `modelV5`| DIN 注意力层内部 MLP 大小，用于计算 Query 和 Keys 相关性得分。 |

### 3. 业务与机制固定参数 (Fixed/Hardcoded Parameters)
与底层物理形态、数学原理深度绑定，**不应修改**。

| 参数名 | 当前设定值 | 所在文件 | 固定的原因与底层逻辑 |
| :--- | :--- | :--- | :--- |
| `num_fields (AFM)` | 4 | `modelV5`| 输入 AFM 的特征域数量。代码严格输入：场、货、人、行为 4 类。 |
| `gamma (GateNU)` | 2.0 | `modelV5`| 门控网络放大系数。将 Sigmoid 的 `[0,1]` 强行拉伸至 `[0,2]`，赋予模型放大(>1)和抑制(<1)双重调权能力。 |
| `padding_idx (Emb)`| 0 | `modelV5`| 词表零值占位符。确保补齐符 0 查出来的向量永远是全零，不产生额外梯度。 |
| `seq_len` | 50 | 数据处理侧 | 用户行为序列的最大截断长度。 |
| `final_proj_out` | 1 | `modelV5`| 塔的最后一层必须映射为 1 维标量(Logit)，随后接入 Sigmoid。 |

| 模块 / 机制 | 最初版框架 (V1/V2) | 当前巅峰框架 (V6) | 带来何种业务增益 |
