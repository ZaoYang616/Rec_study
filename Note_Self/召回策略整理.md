# 向量化召回 (Embedding Retrieval) 之 Item2Vec

## 1. I2I 召回
* **核心任务**：给定一个物品 $i$，如何快速找出与它最“相似”的物品集合？
* **相似度的定义**：
    *  **传统误区**：认为相似度取决于物品的**内在属性**（如：都是红色、都是纯棉）。
    *  **算法视角**：相似度由**用户行为 (User Behavior)** 定义。
        > *"如果两个商品经常被同一批用户购买，或者两部电影经常被同一群观众喜欢，那么它们之间就存在强关联。"*

* **理论来源**：
这一思想源于语言学中的 **Distributional Hypothesis (Firth, 1957)**：
*"You shall know a word by the company it keeps."* > (观其伴，知其义)
即：一个词（或物品）的含义，是由它周围的上下文（Context）决定的。

---

## 2. 理论基石：Word2Vec

Word2Vec (Mikolov et al., 2013) 的成功建立在一个简单而深刻的假设之上：
> 在相似语境中出现的词语往往具有相似的含义。

通过分析海量文本中词语的共现模式，我们可以为每个词学习一个稠密的向量表示，使得语义相近的词在向量空间中距离更近。

Word2Vec主要包含两种模型架构：
* Skip-Gram和
* CBOW（Continuous Bag of Words）。

Skip-Gram模型通过给定的中心词来预测其周围的上下文词，而CBOW模型则相反，通过上下文词来预测中心词。

### 2.1 模型架构：Skip-Gram
推荐系统中最常使用的是 **Skip-Gram** 模型（相较于 CBOW，它对低频词/长尾物品的处理效果更好）。

* **任务**：给定中心词 $w_t$，预测其上下文窗口内的词 $w_{t+j}$，$j\in [-m,m]$。
* **目标函数 (Softmax)**：
    $$
    P(w_{t+j}|w_t) = \frac{e^{v_{w_{t+j}}^T v_{w_t}}}{\sum_{k=1}^{|V|} e^{v_{w_k}^T v_{w_t}}}
    $$
    * **分子**：$v_{w_{t+j}}^T v_{w_t}$ 是向量点积，代表中心词与目标词的相似度。
    * **分母**：所有词的相似度之和（归一化）。$V$是词汇表。

> **读到这里产生的问题：为什么训练时用点积 (Dot Product) 而不是余弦相似度？**
    >
    > 1.  **梯度与收敛 (核心原因)**：Softmax 中 $e^x$ 对数值范围敏感。余弦相似度范围仅 $[-1, 1]$，差异过小导致概率分布过于平滑，梯度消失，难以收敛；而点积无界（如结果可达 10+），$e^{10}$ 远大于 $e^0$，能迅速拉大正负样本差距，提供强有力的梯度。
    > 2.  **模长的语义信息**：点积保留了向量模长 ($||v||$) 信息。模型可以通过学习让重要或高置信度的词拥有更大的模长，这是额外的一个维度自由度。
    > 3.  **计算效率**：点积本质是矩阵乘法 (MatMul)，GPU 计算效率远高于涉及开方和除法的余弦计算。
    >
    > **结论**：**训练时用点积**（为了梯度和优化），**推理/召回时通常用余弦**（为了消除模长偏差，纯粹比较语义方向）。

### 2.2 核心优化：负采样 (Negative Sampling)
由于词汇表 $|V|$ 通常在百万量级，直接计算 Softmax 分母（遍历全表）计算量过大，因此引入 **负采样**。

* **思想**：将“多分类问题”转化为多个“二分类问题”。
* **优化后的 Loss**：
    $$
    L = \log \sigma(v_{pos}^T v_{center}) + \sum_{i=1}^k \mathbb{E}_{v_{neg} \sim P_n(w)} \log \sigma(-v_{neg}^T v_{center})
    $$
    * **正样本部分（第一部分）**：让中心词$v_{pos}$和真实的上下文$v_{center}$向量越相似越好。即中心词与**真实上下文**的点积越大越好（趋近 1）。
    * **负样本部分（第二部分）**：让中心词$v_{neg}$和随机采样的噪声词$v_{center}$向量越不相似越好。随机采样 $k$ 个**噪声词**，让它们与中心词的点积越小越好（趋近 0）。

其中 $\sigma(x) = \frac{1}{1+e^{-x}}$ 是 sigmoid 函数，$k$ 是负样本数量，$P_n(w)$ 是负采样分布。

**直观解释**：
* 对于**真实的词对**，我们希望增加它们的相似度；
* 对于**随机采样的负样本词对**，我们希望降低它们的相似度。

---

## 3. Item2Vec：从 NLP 到推荐系统的迁移

**Item2Vec (Barkan and Koenigstein, 2016)** 是 Word2Vec 在推荐领域的直接应用。

这一迁移建立在 **分布假说 (Distributional Hypothesis)** 的直接类比之上：

* **在 Word2Vec 中**：我们相信 **“一个词的含义，由它周围的词决定”**。
    * *例子*：“我爱吃**苹果**”和“我爱吃**香蕉**”。“苹果”和“香蕉”经常出现在相同的上下文（“我爱吃...”）中，所以它们是相似的。

* **在 Item2Vec 中**：我们相信 **“一个商品的含义，由同时也买了它的其他商品决定”**。
    * *例子*：很多买 **iPhone** 的人，同时也买了 **AirPods**；很多买 **华为Mate** 的人，同时也买了 **FreeBuds**。
    * *机器的视角*：机器虽然不知道什么是“手机”什么是“耳机”，但它通过大量数据发现：iPhone 总和 AirPods 一起出现，所以它们的向量在空间中离得很近。

### 3.1 核心映射关系 (Mapping)

| NLP 领域 | 推荐系统 (RecSys) |
| :--- | :--- |
| **词 (Word)** | **物品 (Item)** |
| **句子 (Sentence)** | **用户交互历史 (User History)** |
| **词的共现** | **物品被同一用户交互** |

### 3.2 集合 vs 序列
* **区别**：原版 Item2Vec 论文指出，在很多场景（如电商订单）中，同一个“篮子”里的商品没有严格的时间先后顺序。
* **处理**：将用户行为视为**集合 (Set)** 而非序列，忽略空间/时间顺序，集合内两两物品互为 Context。

***这是 Item2Vec 和NLP 最大的区别。***

* 在 NLP 中，顺序很重要：
“狗咬人” vs “人咬狗”，意思完全不同。位置信息（Positional information）很关键。

* 在 Item2Vec（电商场景）中，顺序往往没那么重要：
你先买“手机”再买“手机壳”，或者先买“手机壳”再买“手机”，在表示这两个东西的相关性时，其实没区别。它们都是属于同一个“意图（Intent）”的组合。

所以： 很多时候我们把用户的行为看作一个 Bag of Items（物品袋），在这个窗口内的商品两两之间都有关系，而不强制要求“上文预测下文”。

### 3.3 目标函数
优化目标函数与Word2Vec保持一致：
$$
\mathcal{L} = \sum_{s \in S} \sum_{l_i \in s} \sum_{-m \le j \le m, j \neq 0} \log P(l_{i+j}|l_i)
$$
* $S$：数据集
* $s$：单个用户的交互序列/集合
* $l_i$：中心物品
* $l_{i+j}$：上下文物品（共现物品）
* $m$：上下文窗口大小
* $P(l_{i+j}|l_i)$采用与Word2Vec相同的softmax形式计算

## 4 补充

1.  **为什么不用 User2Vec？**
    * 物品 (Item) 是相对静态的，性质稳定。
    * 用户 (User) 兴趣多变，直接训练 User Embedding 极难收敛且面临冷启动。
    * **通用做法**：训练好 Item Embedding 后，通过对用户历史交互过的 Items 取平均 (Average Pooling) 或加权 (Attention) 来得到 User Embedding。

2.  **序列顺序的重要性**
    * **忽略顺序 (Item2Vec)**：适用于“搭配”场景（买了牙膏通常会买牙刷，谁先谁后不重要）。
    * **保留顺序 (DeepWalk / Sequence Embedding)**：适用于“内容消费”场景（看了《令人心动的offer》第一季，大概率接着看第二季）。如果业务强依赖时间序，不要直接用集合式的 Item2Vec。

3.  **负采样的坑 (Negative Sampling)**
    * **Easy Negative**：随机采样全局未点击的物品。
    * **Hard Negative**：在进阶优化中，我们会尝试采样“曝光未点击”或“高热度但用户未点击”的物品作为负样本，这能迫使模型学习到更细粒度的差异（Discriminative Power）。

4.  **相似度计算**
    * 训练完成后，通常使用 **Faiss** 等向量检索库计算 Top-K 近邻。
    * 常用度量：**Cosine Similarity** (余弦相似度)。

# 向量化召回 (Embedding Retrieval) 之 Airbnb业务导向召回算法

## 1. 核心背景：非标品的挑战
Airbnb 的场景与淘宝/京东有本质不同，这也决定了其算法设计的特殊性：
* **低频高价**：用户不像买日用品那样频繁，每一次预订（Booking）都是极其珍贵的强信号。
* **地域限制强**：用户找房通常限定在同一个城市或市场。
* **目标明确**：Airbnb 需要的不仅仅是“相似性”，而是能够真正促进**最终转化（预订）**的推荐。

---

## 2. 核心创新点 I：面向业务的序列构建

传统的 Word2Vec 只是简单地把所有交互串起来，Airbnb 重新定义了“句子”和“单词”的概念。

### 2.1 会话切分 (Session-based)
系统不再将用户历史一锅端，而是基于 **点击会话(Click Sessions)** 构建序列：
* **规则**：当用户连续点击的时间间隔超过 **30分钟** 时，系统自动切分一个新的会话。
* **目的**：捕捉用户在特定时间段、特定搜索场景下的**连贯意图**。

### 2.2 行为权重差异化
Airbnb引入了重要的业务洞察，即不同动作传达的信号和价值完全不同：
* **洞察**：点击（Click）和预订（Booking）的信号强度完全不同。
* **策略**：在模型训练中，预订行为应当给予极高的权重，因为它代表了用户的最终决策。

---

## 3. 核心创新点 II：全局上下文机制 (Global Context)

这是该模型最精彩的改进。标准的 Skip-Gram 只能看到滑动窗口（Window Size）内的局部上下文，容易忽略长序列末端的最终结果，使得其无法充分利用最终预订这一强烈的正向信号。因此，Airbnb让用户最终预订的房源（booked listing）与序列中的每一个浏览房源都形成正样本对进行训练，无论它们在序列中的距离有多远。

### 3.1 引入“预订房源” ($l_b$)
无论滑动窗口滑到序列的哪个位置，Airbnb 强制让**最终预订的房源 ($l_b$)** 始终作为当前中心词的正样本。

### 3.2 优化目标函数
$$
\mathcal{L} = \sum_{(l, c) \in D_p} \log \frac{1}{1 + e^{-v_c^T v_l}} + \sum_{(l, c) \in D_n} \log \frac{1}{1 + e^{v_c^T v_l}} + \underbrace{\log \frac{1}{1 + e^{-v_{l_b}^T v_l}}}_{\text{Global Context (Booking)}}
$$
* $l$是目标房源（当前用户正在预览的房源），$c$是滑动窗口内的上下文房源，$l_b$是最终预定房源。
* **第一项**：最大化局部上下文的正样本相似度（标准 Skip-Gram）。
* **第二项**：最小化负样本相似度（标准 Skip-Gram）。
* **第三项 (创新)**：**全局上下文**。无论当前中心词 $l$ 距离最终预订 $l_b$ 有多远，都要去拟合它们的关系。
    * *物理意义*：即使用户最早点击的房源（可能只是为了探路），也应该与最终买单的房源在向量空间上有某种联系（如风格、价格档次）。

---

## 4. 核心创新点 III：市场感知的负采样 (Market-Aware Negative Sampling)

### 4.1 传统随机采样的缺陷
如果在训练“巴黎”的房源时，随机采样一个“纽约”的房源做负样本，模型会“偷懒”。它只需要学会区分“城市”就能把 Loss 降得很低，而学不到**同一城市内**房源的细微差别（如：都是巴黎的房，为什么选A不选B？）。

### 4.2 增加同市场负样本
Airbnb 强制要求一部分负样本必须来自**与正样本相同的地理市场**。

**负采样公式改进**：
$$
\sum_{(l, l_{m}^{-}) \in D_{m\_neg}} \log \frac{1}{1 + e^{v_{l_{m}^{-}}^T v_l}}
$$
* $l_{m}^{-}$：来自相同市场的负样本。
* **效果**：这迫使模型在“同一个池子”里找不同，学习到更精细的特征（如装修风格、便利设施、房东评分），而不仅仅是地理位置。

---

## 5. 总结和补充

1.  **把 Business Metric 写进 Loss Function**：
    * 很多时候大家很关注 AUC/Recall，但 Airbnb 直接把“预订 ($l_b$)”作为全局正样本塞进 Loss。这告诉我们：**不要让模型只学“相关性”，要让它学“转化”**。

2.  **Hard Negative**：
    * "Market-Aware Negative Sampling" 本质上就是 **Hard Negative Mining**。
    * 在电商里，可以采样“同类目但未点击”的商品；在视频推荐里，可以采样“曝光但被快速划走”的视频。让模型由易到难地学习，才能具备更强的排序能力。

3.  **冷启动处理 (Cold Start)**：
    * Airbnb 的论文中还提到了利用 Listing 的属性（价格、类型、位置）做 Embedding 来解决新房源没有行为数据的问题。对于新物品，利用属性相似物品的平均 Embedding 进行初始化是非常实用的 Trick。

# 向量化召回 (Embedding Retrieval) 之 阿里 EGES (融合辅助信息的图嵌入)/GCS

## 1. 背景：Item2Vec 的局限性
虽然 Item2Vec 验证了序列建模的可行性，但在阿里的十亿级商品场景下，简单的 Item2Vec 面临两个致命问题：
1.  **数据稀疏性 (Sparsity)**：长尾商品交互很少，无法学习到高质量的 Embedding。
2.  **冷启动 (Cold Start)**：新上架的商品完全没有交互记录，无法生成向量。

**EGES** 正是为了解决这两个问题，核心思路是：**引入辅助信息 (Side Information)**。

---

## 2. 基础构建：基于会话的图 (Item Graph)

EGES 首先需要构建训练序列，它采用了**图 (Graph)** 的方式，而非简单的线性序列。

### 2.1 构图策略

具体构建过程如下图所示：当两个商品在同一会话（时间窗口内）的用户行为序列中连续出现时，在它们之间建立一条有向边，边的权重等于这种商品转移模式在所有用户行为历史中出现的频率。相比于传统方法将整个用户历史视为一个序列，这种基于会话的图构建方法能够更准确地捕捉用户在特定时间段内的连续兴趣转移模式。

<div align="center">
  <img src="https://datawhalechina.github.io/fun-rec/_images/eges_item_graph.png" width="600" />
</div>

**快速掌握：**
* **时间窗口**：设定 **1小时** 的时间窗口，只选择该窗口内的用户行为来构建图。
* **边的定义**：如果两个商品在同一会话中连续出现，则建立一条有向边。
* **边的权重**：等于这种跳转在所有用户历史中出现的频率。

### 2.2 带权随机游走 (Weighted Random Walk)
在构建好的图上，使用随机游走生成序列。转移概率由边的权重决定：

$$
P(v_j|v_i) = \begin{cases} 
\frac{M_{ij}}{\sum_{j=1}^{|N_+(v_i)|} M_{ij}} & \text{if } v_j \in N_+(v_i) \\
0 & \text{if } e_{ij} \notin E 
\end{cases}
$$

* $M_{ij}$：节点 $v_i$ 到 $v_j$ 的边权重。
* $N_+(v_i)$：节点$v_i$ 的所有出边邻居。
* **作用**：通过游走生成大量序列，为后续训练提供样本数据。

---

## 3. 进阶 I：GES (Graph Embedding with Side Information)

前面提到的这种纯粹基于行为序列的方法面临一个关键挑战：对于用户交互稀少的商品，由于缺乏足够的共现信息，很难学习到高质量的Embedding表示。为了解决稀疏性，第一步改进是 **GES**。

### 3.1 核心思想
既然 ID 向量学不好（因为数据少），那就利用商品的**固有属性**（品牌、类目、价格等）来增强向量表示。这些属性是稠密的（很多商品共享同一个品牌）。

### 3.2 关键改进：平均融合 (Average Pooling)
将商品本身的 ID Embedding ($W_v^0$) 与其 $n$ 个属性的 Embedding ($W_v^s$) 进行简单平均：

$$
H_v = \frac{1}{n+1} \sum_{s=0}^{n} W_v^s
$$

* **局限性**：它假设所有属性的重要性是一样的。这显然不符合常理（买手机看品牌，买抽纸看价格）。

---

## 4. 终极形态：EGES (Enhanced GES)

EGES的核心创新在于认识到不同类型的辅助信息应该有不同的重要性。对于手机，品牌可能比价格更重要；对于日用品，价格可能比品牌更关键。为了解决 GES“平均主义”的问题，引入了 **注意力机制 (Attention Mechanism)**。

<div align="center">
  <img src="https://datawhalechina.github.io/fun-rec/_images/eges_model.png" width="400" />
</div>

### 4.1 动态权重 (Attention)
EGES 为每个属性引入了一个可学习的权重参数 $a_v^j$。最终的聚合向量 $H_v$ 变为加权和：

$$
H_v = \frac{\sum_{j=0}^{n} e^{a_v^j} W_v^j}{\sum_{j=0}^{n} e^{a_v^j}}
$$

* $e^{a_v^j}$：使用指数函数确保权重非负，并作为 Attention Score。
* $W_v^j$：第 $j$ 个特征（包含 ID 和 Side Info）的 Embedding。

### 4.2 物理意义
这种设计让模型能够自动感知不同商品对属性的依赖程度：
* 对于 **手机**：模型会自动给 **“品牌”** 赋予更高的权重 $a$。
* 对于 **日用品**：模型会自动给 **“价格”** 赋予更高的权重 $a$。

---

## 5. 冷启动策略 (Cold Start)

对于完全没有交互历史的新商品（没有 ID Embedding，也没有训练好的权重 $a$）：

* **策略**：采用简单的 **Mean Pooling**。
* **操作**：直接对该商品的所有辅助信息向量（类别、品牌、价格等）取平均。
* **效果**：虽然不如 EGES 精细，但它利用内容特征保证了新商品至少能被召回，实现了从 0 到 1 的突破。

---
### 6. EGES 的损失函数 (Loss Function)

尽管 EGES 引入了复杂的辅助信息融合和 Attention 机制，其训练目标依然源自 Word2Vec 的 **Skip-Gram 模型**，并采用了 **负采样 (Negative Sampling)** 进行优化。

#### 6.1 目标函数公式
我们需要最大化正样本（随机游走序列中相邻的商品）的共现概率，同时最小化负样本（随机采样商品）的共现概率。

形式化地，对于每一个训练样本对 $(u, v)$（其中 $u$ 是中心节点，$v$ 是游走序列中的上下文节点），损失函数定义为：

$$
\mathcal{L} = - \sum_{(u, v) \in D} \left( \log \sigma(H_u^T H_v) + \sum_{k=1}^K \mathbb{E}_{v_{neg} \sim P_n} [\log \sigma(-H_u^T H_{v_{neg}})] \right)
$$
* **$D$**：正样本集合（通过带权随机游走生成的序列对）。
* **$H_u, H_v$**：**注意这里是核心区别！** 在传统的 Word2Vec 中，这里直接是查表得到的 ID 向量；而在 EGES 中，这里是**经过 Attention 加权融合后的聚合向量**（即上文提到的 $H_v = \frac{\sum e^{a^j} W^j}{\sum e^{a^j}}$）。
* **$\sigma(\cdot)$**：Sigmoid 激活函数。
* **$K$**：负采样个数。

#### 6.2 梯度反向传播 (Backpropagation)
这个损失函数的精妙之处在于它的**端到端 (End-to-End)** 学习能力。当我们对 $\mathcal{L}$ 求导并反向传播时，梯度会流向两个部分：

1.  **流向 Embeddings ($W^j$)**：更新商品 ID 的 Embedding 以及 品牌、价格等辅助信息的 Embeddings。
2.  **流向 Attention 权重 ($a^j$)**：**这是最关键的**。模型会根据 Loss 的反馈，自动调整 $a^j$ 的值。
    * 如果“品牌”特征对区分正负样本很有帮助，梯度会推高品牌的权重 $a_{brand}$。
    * 如果“颜色”特征是噪声，梯度会降低颜色的权重 $a_{color}$。

---

> **通俗理解**
>
> 把 EGES 想象成一个**“特征加权器”**外挂了一个**“Skip-Gram 训练器”**。
> * **Skip-Gram 说**：“我不管你传给我的是什么，我只负责拉近经常在一起出现的向量，推远不在一起的。”
> * **EGES 层说**：“好的，那我负责把 ID、品牌、价格拼成一个最好的向量 $H$ 传给你。”
>
> 整个训练过程，就是通过 Skip-Gram 的任务压力，倒逼 EGES 层去学会到底该信哪个特征。

---

## 7. 总结和补充

1.  **Embedding 的本质是降维，也是一种“泛化”**：
    * 单纯使用 Item ID 是死记硬背的体现。
    * Side Information (比如Brand/Category) 能够做到举一反三。EGES 完美结合了两者。

2.  **关于 Attention 的工程细节**：
    * 注意公式里的 $e^{a_v^j}$。在实际代码实现时，通常直接用 Softmax 函数对 $a$ 进行归一化，效果是一样的。
    * 训练时，不仅更新 Embedding $W$，也在更新权重 $a$。这意味着**每个商品都有自己的一套属性权重偏好**。

3.  **图 (Graph) 还是 序列 (Sequence)？**
    * 这里虽然叫 Graph Embedding，但本质上还是 `Random Walk -> Sequence -> Skip-Gram` 的路径。
    * 构建 Graph 的主要目的是为了更好地从**全局**角度挖掘 item 之间的共现关系（通过边权重 $M_{ij}$），比单纯切分 Session 能包含更多统计信息。

# 总览

**定义转变**：从“物”到“人”
* **I2I 解决的问题**：“买了**这个商品**的人，还会买什么？”（侧重物品的相关性）。
* **U2I 解决的问题**：“**这个用户**会喜欢什么商品？”（侧重用户的个性化兴趣）。

U2I 面临的最大挑战在于**计算复杂度**。在面对数亿用户和千万级商品库时，传统的协同过滤（矩阵分解）计算复杂度过高，难以实时响应。

为了解决上述挑战，工业界演化出了一套统一的架构思想——**双塔模型**。无论是早期的 DSSM，还是经典的 YouTubeDNN，本质上都遵循这一范式。

总的来说，双塔模型将推荐问题分解为两个相对独立的子问题（两个塔）：
1.  **用户塔 (User Tower)**：
    * **输入**：用户的历史行为、人口统计学特征（年龄/性别）、上下文信息。
    * **输出**：一个代表用户兴趣的**用户向量 $u$**。

2.  **物品塔 (Item Tower)**：
    * **输入**：物品ID、类别、属性、内容特征。
    * **输出**：一个代表物品特性的**物品向量 $v$**。

**匹配度量定义：**
用户与物品的契合度，通过两个向量的**点积 (Dot Product)** 或 **余弦相似度** 来衡量：

$$
\text{score}(u, v) = u \cdot v = \sum_{i=1}^{d} u_i v_i
$$

其中 $d$ 是向量维度。这个简单的数学操作背后隐含着深刻的物理意义：**向量空间中的距离反映了用户兴趣与物品特性的匹配程度**。

**为什么现在主要用双塔？**
双塔结构之所以成为主流，最大的原因在于其**分而治之**带来的巨大工程优势。

* **离线计算 (Offline)**：
    * 所有物品的向量 $v$ 可以**预先计算**好，并存入向量检索引擎（如 **Faiss**, **Annoy**, **Milvus**）。
* **在线服务 (Online)**：
    * 当用户发起请求时，系统只需实时计算当前的一个用户向量 $u$。
    * 然后利用 **ANN (Approximate Nearest Neighbor, 近似最近邻搜索)** 技术，在毫秒级时间内从海量物品库中找出与 $u$ 最相似的 Top-K 个物品 $v$。

**复杂度降维**：
这种架构巧妙地将原本需要 $O(U \times I)$ 的复杂匹配问题，降低到了 $O(U + I)$ 的量级（配合 ANN 甚至更快），使得在海量候选集中进行实时推荐成为可能。

**面临的问题**
1.  **速度与精度的权衡**：
    * 双塔要求用户特征和物品特征在最后一步点积之前**不能互斥**。这意味着我们无法使用“用户行为x物品属性”这样的交叉特征（Cross Features）。
    * *代价*：模型对细节的捕捉能力不如精排模型（Ranking）。
    * *收益*：换来了极致的检索速度，这对于**召回阶段**是至关重要的。

2.  **Serving**：
    * 物品塔通常是一天更新一次（T+1），或者小时级更新。
    * 用户塔必须是**实时**的（Real-time），因为用户的兴趣随点击行为在秒级变化。

# 向量化召回 (Embedding Retrieval) 之 FM（因子分解机）

## 1. 核心定位：从“记忆”到“泛化”
* **相比 LR (线性回归)**：FM 引入了**二阶特征交互**（Feature Interaction），解决了 LR 无法捕获“啤酒+尿布”这种组合特征的问题。
* **相比 Collaborative Filtering (协同过滤)**：FM 能够处理稀疏数据，且可以无缝融合内容特征。

FM 首次将复杂的特征交互，优雅地分解为两个低维向量的内积操作，这本质上就是**双塔模型的雏形**。

---

## 2. 数学原理

### 2.1 从交互矩阵到向量内积
FM 模型的目标是预测评分或点击率。其核心公式为：

$$
\hat{y}(\mathbf{x}) := \underbrace{w_0}_{\text{Global Bias}} + \underbrace{\sum_{i=1}^n w_i x_i}_{\text{Linear Terms}} + \underbrace{\sum_{i=1}^n \sum_{j=i+1}^n \langle \mathbf{v}_i, \mathbf{v}_j \rangle x_i x_j}_{\text{Interaction Terms}}
$$

* 每个特征 $i$ 都对应一个 $k$ 维的隐向量 $\mathbf{v}_i$，特征间的交互通过这些隐向量的内积 $\langle \mathbf{v}_i, \mathbf{v}_j \rangle = \sum_{f=1}^k v_{i,f} \cdot v_{j,f}$ 来建模。
* **问题**：第三项（交叉项）的直接计算复杂度是 $O(n^2)$，在特征维度 $n$ 很大时（如亿级 ID）根本无法计算。
* **核心 Trick**：利用数学恒等式 $(a+b)^2 - a^2 - b^2 = 2ab$，将计算复杂度从 $O(kn^2)$ 降维到 $O(kn)$。

$$
\sum_{i=1}^n \sum_{j=i+1}^n \langle \mathbf{v}_i, \mathbf{v}_j \rangle x_i x_j = \frac{1}{2} \sum_{f=1}^k \left( \left( \sum_{i=1}^n v_{i,f} x_i \right)^2 - \sum_{i=1}^n v_{i,f}^2 x_i^2 \right)
$$

>  **补充：如何直观理解这个数学变换？**
>
> **1. 直觉推导：矩阵面积法**
> 假设只要计算 3 个特征交互 $(ab + bc + ac)$，可以利用初中数学公式 $(a+b+c)^2 = a^2+b^2+c^2 + 2ab+2bc+2ac$。
> 想象一个 $3 \times 3$ 的矩阵，其中第 $i$ 行第 $j$ 列的元素是 $x_i v_i \cdot x_j v_j$：
> * **总面积**：整个大正方形的面积，即 $(\sum x_i v_i)^2$。
> * **对角线**：自相关部分 $x_i^2 v_i^2$，这是我们不需要的，需**扣除**。
> * **交叉项**：剩下的部分是两个三角形（左下+右上）。因为矩阵是对称的，这两个三角形面积相等。
>
> **结论**：我们要算的 FM 交叉项（也就是上三角部分），等于 **(总面积 - 对角线) / 2**。这就是公式里 $\frac{1}{2}[(\dots)^2 - \dots^2]$ 的物理含义
>
> ---
>
> **2. 严谨推导**
> 如果要把 Embedding 的维度 $f$ (比如 64 维) 加回去，推导过程如下：
>
> * **Step 1: 展开内积**
>     内积 $\langle \mathbf{v}_i, \mathbf{v}_j \rangle$ 本质是 $k$ 个维度相乘再相加。带入目标公式：
>     $$\sum_{i=1}^n \sum_{j=i+1}^n \left( \sum_{f=1}^k v_{i,f} v_{j,f} \right) x_i x_j$$
>
> * **Step 2: 交换求和顺序**
>     既然都是加法，可以先把 $f$ (维度) 提到最外面来算：
>     $$\sum_{f=1}^k \left[ \sum_{i=1}^n \sum_{j=i+1}^n (v_{i,f} x_i) (v_{j,f} x_j) \right]$$
>     *注意中括号里的部分就是前面说的标量版 $ab+bc+ac$*
>
> * **Step 3: 套用完全平方公式**
>     对中括号部分使用 $\frac{1}{2}[(\text{和})^2 - \text{平方和}]$ 的变换：
>     $$\text{中括号部分} = \frac{1}{2} \left[ \left( \sum_{i=1}^n v_{i,f} x_i \right)^2 - \sum_{i=1}^n (v_{i,f} x_i)^2 \right]$$
>
> * **Step 4: 最终合并**
>     把外面的 $\sum_{f=1}^k$ 放回来，提取常数 $\frac{1}{2}$，就得到了那个看起来很复杂的最终公式。
>
> **工程意义**：
> 左边是双重循环求和 $\sum_{i=1}^n \sum_{j=i+1}^n$ 依然需要算 $n(n-1)/2$ 次。 $O(kn^2)$，线上服务根本跑不动；
> 右边变成了单重循环 $O(kn)$，先算所有 $v_i x_i$ 的和，再平方。这使得 FM 具备了处理**亿级稀疏特征**的实战能力。

---

## 3. 核心推导：FM 的双塔拆解

虽然FM通过数学变换解决了计算复杂度问题，但在召回任务中，我们还面临另一个挑战：如何高效地为用户从海量候选物品中筛选出最相关的推荐结果？这时就需要考虑将FM分解为双塔结构。

我们通常认为 FM 只能做排序（因为要算交叉），但通过公式重组，我们可以把它强行拆成 **User Vector** 和 **Item Vector**，从而支持 ANN 检索。

### 3.1 特征分组
我们将特征分为两大类：
* **用户侧特征集 $U$**：用户 ID、年龄、历史行为等。
* **物品侧特征集 $I$**：物品 ID、类目、店铺等。

### 3.2 交叉项的拆解

**这里有一个关键的发现**：当我们为同一个用户推荐不同物品时，用户特征是固定不变的。因此，用户特征内部的交互得分（无论是一阶还是二阶）对所有候选物品都是相同的。既然这部分得分相同，在排序时就可以忽略，我们只需要关注： 1. 物品特征内部的交互得分 2. 用户特征与物品特征之间的交互得分

**快速总结**，即：
当我们针对一个特定用户（特征 $U$ 固定）对海量物品（特征 $I$ 变化）进行打分时，FM 的二阶项可以展开为三部分：

1.  **用户内部交互 ($U \times U$)**：$\sum_{u \in U} \sum_{u' \in U} \dots$
    * *分析*：对于同一个用户，无论推哪个物品，这项得分都是**常数**。**在排序时可以忽略！**
2.  **物品内部交互 ($I \times I$)**：$\sum_{i \in I} \sum_{i' \in I} \dots$
    * *分析*：这部分只和物品有关。可以**离线预计算**并在物品上线时存入属性中。
3.  **用户-物品交叉 ($U \times I$)**：$\sum_{u \in U} \sum_{i \in I} \dots$
    * *分析*：这是**决定个性化匹配**的核心项。

基于这个思路，我们可以将 FM 的二阶交互项重新组织：

$$
\begin{aligned}
& \frac{1}{2} \sum_{f=1}^k \left( \left( \sum_{i=1}^n v_{i,f} x_i \right)^2 - \sum_{i=1}^n v_{i,f}^2 x_i^2 \right) \\
= & \frac{1}{2} \sum_{f=1}^k \left( \left( \sum_{u \in U} v_{u,f} x_u + \sum_{t \in I} v_{t,f} x_t \right)^2 - \sum_{u \in U} v_{u,f}^2 x_u^2 - \sum_{t \in I} v_{t,f}^2 x_t^2 \right) \\
= & \frac{1}{2} \sum_{f=1}^k \left( \left( \sum_{u \in U} v_{u,f} x_u \right)^2 + \left( \sum_{t \in I} v_{t,f} x_t \right)^2 + 2 \sum_{u \in U} v_{u,f} x_u \sum_{t \in I} v_{t,f} x_t - \sum_{u \in U} v_{u,f}^2 x_u^2 - \sum_{t \in I} v_{t,f}^2 x_t^2 \right)
\end{aligned}
$$

基于前面的分析，我们发现**用户特征内部的交互项对所有候选物品都相同**（即上式中仅包含 $U$ 的项），因此可以在召回阶段忽略。这样就可以将 FM 重新组织，只保留对排序有影响的部分：

$$
\text{score}_{FM} = \sum_{t \in I} w_t x_t + \frac{1}{2} \sum_{f=1}^k \left( \left( \sum_{t \in I} v_{t,f} x_t \right)^2 - \sum_{t \in I} v_{t,f}^2 x_t^2 \right) + \sum_{f=1}^k \left( \sum_{u \in U} v_{u,f} x_u \sum_{t \in I} v_{t,f} x_t \right)
$$

观察上面的公式，我们发现一个重要的数学结构：最后一项 $\sum_{f=1}^k (\dots)$ 实际上是两个向量 $\sum_{u \in U} v_u x_u$ 和 $\sum_{t \in I} v_t x_t$ 的内积。这启发我们将整个匹配分数重新组织为双塔结构：

$$
\text{score}_{FM} = V_{item} \cdot V_{user}^T
$$

通过这种重新组织，我们得到了 FM 的双塔表示：

* **用户向量**：$V_{user} = [1; \sum_{u \in U} v_u x_u]$
* **物品向量**：$V_{item} = [\sum_{t \in I} w_t x_t + \frac{1}{2} \sum_{f=1}^k ((\sum_{t \in I} v_{t,f} x_t)^2 - \sum_{t \in I} v_{t,f}^2 x_t^2); \sum_{t \in I} v_t x_t]$

这里的**设计很巧妙**：
* **用户向量**包含一个常数项 `1` 和用户特征的聚合表示。
* **物品向量**则包含物品的内部交互信息（作为第一维）和物品特征的聚合表示。
* 两个向量通过**内积运算**，既能捕捉用户-物品之间的交互（User-Item Interaction），又保留了物品内部特征的复杂关系（Item-Item Interaction）。

这样的分解揭示了一个重要原理：即使是复杂的特征交互模式，也可以通过合适的向量表示和简单的内积运算来实现。

### 3.3 最终的双塔形式
经过重组，FM 的打分公式可以重写为两个向量的内积：

$$
\text{score}_{FM} = V_{user}^T \cdot V_{item}
$$

**构造方式如下**：

| 向量分量 | User Vector ($V_{user}$) | Item Vector ($V_{item}$) |
| :--- | :--- | :--- |
| **第一部分 (偏置/常数)** | $1$ | $\sum_{t \in I} w_t x_t + \text{Item\_Internal\_Interaction}$ |
| **第二部分 (Latent Interaction)** | $\sum_{u \in U} v_u x_u$ | $\sum_{t \in I} v_t x_t$ |

* **$V_{user}$**：由一个常数 `1` 和用户所有特征 Embedding 的和组成。
* **$V_{item}$**：由物品的线性分+内部交叉分，以及物品所有特征 Embedding 的和组成。

> **几何意义**：
> 这种构造巧妙地把“线性项”和“二阶项”打包进了一个统一的内积空间。当你做点积时：
> * $1 \times (\dots)$ 这一项还原了物品的 Bias 和内部交互。
> * $(\sum v_u) \times (\sum v_i)$ 这一项还原了用户和物品的 Cross Interaction。

## 4. 代码实现 (伪代码)

基于上述推导，我们可以轻松写出支持离线索引的 FM 召回模型。

```python
# 假设 batch_size=B, embedding_dim=K

# === 1. 构建用户塔 (Online Calculation) ===
# 用户 Embedding 求和
user_emb_sum = SumPooling(user_embeddings)  # Shape: [B, K]
# 构造常数项 1
ones = Ones((B, 1))                         # Shape: [B, 1]
# 拼接得到 User Vector
V_user = Concat([ones, user_emb_sum])       # Shape: [B, K+1]

# === 2. 构建物品塔 (Offline Calculation) ===
# A. 物品线性项 (w * x)
item_linear = Linear(item_features)         # Shape: [B, 1]

# B. 物品内部交互项 (0.5 * [(sum)^2 - sum(^2)])
item_emb_sum = SumPooling(item_embeddings)
item_emb_sq  = Square(item_embeddings)
sum_sq = SumPooling(item_emb_sq)
item_internal_interaction = 0.5 * (Square(item_emb_sum) - sum_sq) # 简化版示意，通常取sum

# C. 组合物品向量的第一部分 (Scalar Part)
# 这里将线性项和内部交互项合并为一个标量
scalar_part = item_linear + ReduceSum(item_internal_interaction) 

# D. 拼接得到 Item Vector
V_item = Concat([scalar_part, item_emb_sum]) # Shape: [B, K+1]

# === 3. 召回匹配 (ANN Retrieval) ===
# score = V_user dot V_item
# 实际部署时，V_item 存入 Faiss，V_user 实时计算并检索
```

## 5. 总结和补充

1.  **FM 可以在召回和排序中使用，但是用法不同**
    * **在精排 (Ranking) 阶段**：我们完全使用原始公式（考虑所有交互），保留所有交叉项，以追求最高精度。
    * **在召回 (Retrieval) 阶段**：我们使用**双塔拆解公式**，牺牲掉用户内部特征交互（因为对于同一用户它是常数，不影响排序），换取了 **$O(1)$** 的极速检索能力。

2.  **维度对齐需要注意**
    * 注意 $V_{user}$ 的第一维是 **常数 1**。
    * 注意 $V_{item}$ 的第一维是 **复杂的标量**（线性项 + 内部交互项）。
    * **实战注意**：在使用 **Faiss** 等向量检索引擎时，维度必须严格对齐。例如：如果 Embedding 是 64 维，那么存入 Faiss 的向量总维度应为 **65 维**（1维 Bias + 64维 Embedding）。

3.  **为什么现在很少单独用 FM 做召回？**
    * **局限性**：FM 只能做**线性**和**二阶**交互。
    * **现状**：现代的双塔模型（如 DSSM, YouTubeDNN）通过 **MLP (多层感知机)** 可以拟合任意高阶的非线性关系，表达能力更强。
    * **价值**：但 FM 的思想是永恒的——**Embedding + Interaction + Decomposition**（嵌入、交互、分解），这是所有现代推荐算法的基石。

# 向量化召回 (Embedding Retrieval) 之 DSSM

## 1. 定位转变：从“线性”到“深度”

虽然 FM 通过数学技巧实现了向量内积的召回形式，但其本质仍然是浅层的线性模型（Linear Model），难以捕捉复杂的非线性特征交互。

**DSSM (Deep Structured Semantic Model)** 的出现，通过深度神经网络替代线性变换，实现了更强的特征学习和表示能力。
* **核心思想**：利用深度神经网络 (DNN) 强大的拟合能力，将用户 (User) 和物品 (Item) 映射到同一个**共享的语义空间**中。
* **度量方式**：在语义空间中，通过计算两个向量的余弦相似度或点积来衡量用户与物品的匹配程度。

---

## 2. 模型架构：独立的双塔

在推荐系统中，DSSM 的架构非常清晰，分为两个独立的子网络（塔）：用户塔和物品塔，每个塔都是独立的DNN结构。用户特征（如历史行为、人口统计学信息等）经过用户塔处理后输出用户Embedding，物品特征（如ID、类别、属性等）经过物品塔处理后输出物品Embedding。两个Embedding的维度必须保持一致，以便进行后续的相似度计算。

<div align="center">
  <img src="https://datawhalechina.github.io/fun-rec/_images/dssm_architecture.svg" width="400" />
</div>

### 2.1 结构拆解
1.  **用户塔 (User Tower)**：
    * **输入**：用户画像（年龄、性别）、行为序列、上下文特征。
    * **处理**：经过 Embedding 层和多层 MLP (ReLU/Tanh) 的非线性变换。
    * **输出**：用户 Embedding $u$。
2.  **物品塔 (Item Tower)**：
    * **输入**：物品 ID、类目、文本描述、图像特征。
    * **处理**：同样经过 Embedding 和 MLP 变换。
    * **输出**：物品 Embedding $v$。
3.  **交互层 (Interaction)**：
    * **特点**：两个塔在最后一步之前**互不干扰**。
    * **计算**：仅在输出层计算 $u$ 和 $v$ 的内积或余弦相似度。

---

## 3. 训练范式：极端多分类问题

DSSM 将召回任务视为一个**极端多分类问题 (Extreme Multi-class Classification)**。将物料库中的所有物品看作不同的类别。模型的目标是最大化用户对正样本物品的预测概率：

### 3.1 概率定义
模型的目标是最大化用户 $x$ 点击正样本物品 $y$ 的概率：

$$
P(y|x, \theta) = \frac{e^{s(x, y)}}{\sum_{j \in M} e^{s(x, y_j)}}
$$

* $s(x, y)$：用户 $x$ 和物品 $y$ 的相似度打分（通常是点积）。
* $M$：整个物品库（候选集）。

### 3.2 负采样 (Negative Sampling)
由于分母需要遍历海量物品库（百万/千万级），直接计算 Softmax 是不可行的。
**解决方案**：采用负采样技术。对于每个正样本，随机采样一定数量的负样本来近似计算分母。

---

## 4. 关键细节

除了网络结构，双塔模型在落地时有两个极易被忽视但至关重要的细节。

### 4.1 向量归一化 (Normalization)
在最后输出 Embedding 时，通常会对 $u$ 和 $v$ 进行 **L2 归一化**：
$$
u \leftarrow \frac{u}{||u||_2}, \quad v \leftarrow \frac{v}{||v||_2}
$$

**Q：为什么要这样做？**
1.  **解决非度量性问题**：原始点积不满足三角不等式，可能导致距离计算的不一致。
2.  **训练与检索的一致性 (关键)**：
    * *线上服务 (Serving)*：为了加速，我们通常使用 **ANN (如 Faiss)**，其常用的距离度量是**欧式距离**。
    * *数学推导*：对于归一化向量，欧式距离与点积存在单调关系：
        $$
        ||u - v|| = \sqrt{2 - 2\langle u, v \rangle}
        $$
    * **结论**：归一化保证了训练时的“相似度最大化”等价于线上检索时的“欧式距离最小化”，避免了训练和上线效果不一致的问题。

### 4.2 温度系数调节 (Temperature Scaling)
在计算 Softmax 之前，除以一个温度系数 $\tau$：
$$
s(u, v) = \frac{\langle u, v \rangle}{\tau}
$$

**$\tau$ 的作用**：
* **$\tau < 1$**：差异被放大，概率分布变得尖锐（Sharp）。模型对高分样本更“自信”，但也更容易过拟合。
* **$\tau > 1$**：差异被缩小，概率分布变得平滑（Smooth）。
* **工业经验**：$\tau$ 是一个超参数，实际上是在调节 Logits 的数值范围，防止 Softmax 进入梯度饱和区，对收敛至关重要。

---

## 5. 总结与补充

1.  **双塔的“不可能三角”**
    * DSSM 最大的优势是 **工程解耦**（物品向量可离线存，用户向量实时算）。
    * 最大的劣势是 **无法进行早期特征交叉**。用户特征和物品特征直到最后一步才见面。
    * *推论*：这意味着 DSSM 无法捕捉形如“<用户是男性> 且 <物品是连衣裙>”这种显式的交叉特征。这也是为什么召回层之后必须要有精排层（Ranking）的原因。

2.  **关于温度系数 $\tau$ 的设置**
    * 这是实战中需要注意的问题。如果不加 $\tau$，直接用归一化后的点积（范围 -1 到 1）输入 Softmax，数值太小，导致概率分布接近均匀分布（Entropy高），模型很难学。
    * 通常 $\tau$ 设置在 $0.07 \sim 0.2$ 之间，人为地把 Logits 拉大，强迫模型做出选择。

3.  **负采样问题**
    * Sampled Softmax 假设负采样是均匀的。但如果你为了提升难度使用了“热门负采样”，在计算 Logits 时需要进行 **LogQ 修正**，否则模型会倾向于过度打压热门商品。

# 向量化召回 (Embedding Retrieval) 之 YouTubeDNN

## 1. 核心定义：从“匹配”到“预测下一刻”

YouTubeDNN 并没有简单地将召回定义为“相似度匹配”，而是将其重新定义为一个 **“预测用户下一个会观看的视频”** 的问题。

* **类比 NLP**：这本质上非常类似于 NLP 中的 **Next Token Prediction**（给定上文单词预测下一个单词）。
* **问题建模**：这是一个**极端多分类问题 (Extreme Multiclass Classification)**。对于时刻 $t$，给定用户历史 $U$ 和上下文 $C$，预测在海量视频库 $V$（百万/千万级）中，用户观看视频 $i$ 的概率。

$$
P(w_t = i | U, C) = \frac{e^{v_i u}}{\sum_{j \in V} e^{v_j u}}
$$

* $u \in \mathbb{R}^N$：用户 Embedding（由神经网络生成）。
* $v_i \in \mathbb{R}^N$：视频 $i$ 的 Embedding。

---

## 2. 模型架构：非对称的双塔

与 DSSM 的对称双塔不同，YouTubeDNN 采用了一种**非对称**的架构设计。

<div align="center">
  <img src="https://datawhalechina.github.io/fun-rec/_images/youtubednn_candidate.png" width="500" />
</div>

### 2.1 用户塔 (复杂且深)
用户侧特征极其丰富，通过多层 ReLU 网络进行特征提取和压缩：
* **观看历史 (Watch Vector)**：用户看过的视频 ID 序列，通过 Embedding 映射后取**平均 (Average Pooling)**。
* **搜索历史 (Search Vector)**：用户搜索过的关键词 (Token)，同样 Embedding后 **取平均**。
* **人口统计学特征**：地理位置 (Geo)、设备、性别、登录状态等。
* **Example Age**：视频上传时间距离现在的间隔（详见下文工程技巧）。

### 2.2 物品塔 (简化)
在架构图中，不到显式的物品塔。
* **本质**：物品向量 $v_j$ 其实就是 **Softmax 层的权重矩阵 (Weights)**。（后面有解释）
* **训练时**：通过反向传播更新 Softmax 层的权重，这些权重最终成为了物品的 Embedding。
* **服务时**：直接把这些权重取出来，存入向量检索引擎（索引）。

---

## 3. 关键工程技巧 (The "Secret Sauce")

YouTubeDNN 的成功不仅仅在于网络结构，更在于它解决实际工业界痛点的一系列 Trick。

### 3.1 Example Age (对付“新”偏见)
* **问题**：视频推荐有极强的**时效性**（用户喜欢看刚出的新视频）。但机器学习模型本质上是学习过去的数据，倾向于推荐“过去很火”的老视频（长尾效应）。
* **做法**：
    * **训练时**：将 `example_age`（样本产生时间 - 视频上传时间）作为一个数值特征喂给模型。告诉模型：“这个视频已经发布了 $x$ 天了”。
    * **预测时 (Serving)**：**强行将 `example_age` 置为 0**（或者一个很小的负数）。
* **效果**：这相当于欺骗模型：“这个视频是刚刚发布的，非常新鲜！”。模型会根据学到的权重，给予新视频更高的打分，从而人为地boost了新鲜度。

### 3.2 非对称时序切分 (Asymmetric Temporal Split)
* **问题**：传统的随机留一法（Random Holdout）会产生**未来信息泄露**。比如用“周二”的行为去预测“周一”的观看，这在实际中是不可能的。
* **做法**：严格按照时间顺序划分训练集和测试集。使用用户 $t$ 时刻之前的历史来预测 $t$ 时刻的行为。

<div align="center">
  <img src="https://datawhalechina.github.io/fun-rec/_images/youtubednn_temporal_split.png" width="600" />
</div>

### 3.3 负采样与 Sampled Softmax
由于类别数（视频库）高达百万级，分母计算不可行。模型采用了 **Sampled Softmax**（重要性采样），每次训练只随机采样数千个负样本进行梯度更新，将训练速度提升了 100 多倍。

## 4. 核心代码实现

这段伪代码展示了 YouTubeDNN 如何将复杂的特征输入转化为向量，并利用 Sampled Softmax 进行高效训练。

### 4.1 用户塔 (User Tower) —— 特征的大熔炉

```python
# === 1. 特征融合 (Feature Engineering) ===
# 将非序列特征（如年龄、性别、地理位置、Example Age）进行拼接
# group_embedding_feature_dict 包含了所有离散特征的 Embedding 结果
user_feature_embedding = concat_group_embedding(
    group_embedding_feature_dict, "user_dnn"
) # Shape: [Batch_Size, Non_Seq_Feat_Dim]

# 处理序列特征 (Watch History / Search History)
if "raw_hist_seq" in group_embedding_feature_dict:
    # 关键步骤：对变长的历史行为序列进行 Embedding 映射
    # 然后进行 Average Pooling (平均池化)，将其变成定长向量
    # 对应理论：把“观看历史”视为“句子”，取词向量的平均值
    hist_seq_embedding = concat_group_embedding(
        group_embedding_feature_dict, "raw_hist_seq"
    ) # Shape: [Batch_Size, Embedding_Dim]
    
    # 将 基础特征 与 历史行为特征 拼接
    user_dnn_inputs = tf.concat(
        [user_feature_embedding, hist_seq_embedding], axis=1
    ) # Shape: [Batch_Size, Total_Dim]
else:
    user_dnn_inputs = user_feature_embedding

# === 2. 深度网络生成向量 (MLP & Normalization) ===
# 通过多层 ReLU 网络拟合非线性关系
# 对应架构图中的塔状结构 (ReLU -> ReLU -> ReLU)
user_dnn_output = DNNs(
    units=dnn_units + [emb_dim],  # 最后一层维度必须等于 Item Embedding 维度
    activation="relu", 
    use_bn=False
)(user_dnn_inputs)

# 关键 Trick：L2 归一化
# 对应理论：保证训练时的点积相似度与线上 Serving 时的欧式距离一致
user_dnn_output = L2NormalizeLayer(axis=-1)(user_dnn_output)
# 最终输出：User Vector u
```
### 4.2 物品塔 (Item Tower) —— 简单的查表
```python
# === 3. 获取物品向量 ===
# 从全局的大 Embedding 表中查出当前训练样本对应的 Item Vector
# label_name 是正样本物品的 ID
item_embedding_table = embedding_table_dict[label_name]

# 维度调整：移除多余的维度
output_item_embedding = SqueezeLayer(axis=1)(
    item_layer_dict[label_name]
)

# 同样进行 L2 归一化，与 User Vector 保持量纲一致
output_item_embedding = L2NormalizeLayer(axis=-1)(output_item_embedding)
# 最终输出：Item Vector v (即 Softmax 层的权重)
```
### 4.3 训练层 (Training) —— 采样优化
```python
# === 4. 采样 Softmax (Sampled Softmax) ===
# 初始化采样层
# item_vocab_size: 视频库总大小 (e.g., 1000万)
# neg_sample: 每次训练采样的负样本数 (e.g., 2000-5000)
sampled_softmax_layer = SampledSoftmaxLayer(
    item_vocab_size, neg_sample, emb_dim
)

# 计算 Loss
# 输入：
# 1. item_embedding_table: 所有的 Item Embedding (作为 Softmax 的权重矩阵 W)
# 2. user_dnn_output: 用户向量 u (作为查询 Query)
# 3. input_layer_dict[label_name]: 正样本标签 (用于计算梯度)
output = sampled_softmax_layer([
    item_embedding_table.embeddings,
    user_dnn_output,
    input_layer_dict[label_name]
])
```

## 5. 总结和补充

1.  **所有特征皆可 Embedding**
    * YouTubeDNN 展示了如何处理多模态特征：ID 类特征（视频/搜索词）做 Embedding 后 **Average Pooling** 是最简单有效的定长化方法。虽然会损失序列顺序信息，但在召回阶段通常足够了。

2.  **User Bias 的处理**
    * 笔记中提到了 **“用户样本均衡”**。在实战中，活跃用户贡献了绝大多数 Log。如果不做限制，模型会过度拟合活跃用户的喜好，忽略低频用户。
    * *Trick*：限制每个用户在训练集中能贡献的样本上限（例如每个用户最多取最近 50 条 Log）。

3.  **Softmax 也就是向量内积**
    * 为什么说 Softmax 层权重就是 Item Embedding？
    * Softmax 的 logits 计算公式是 $W^T x + b$。这里 $x$ 是 User Embedding。那么 $W$ 的每一行显然就是对应类别的 Item Embedding。这是双塔模型最精简的实现形式。
