# 推荐系统序列建模：Target Attention vs Self-Attention

**核心命题**：为什么在召回阶段的双塔模型中，Target Attention 是绝对的“禁忌”，而目前主流都在使用 Self-Attention？

---

## 1. 核心痛点：为什么双塔模型不能用 Target Attention？

一句话总结：**Target Attention 破坏了双塔模型的“彻底解耦”前提，会导致线上计算量爆炸，完全无法使用 Faiss 等向量检索工具。**

### 1.1 双塔模型的底层逻辑 (Late Crossing)
* **工程要求**：线上召回必须在几十毫秒内，从千万级商品库中找回 Top 1000。
* **实现方式**：**离线**预计算好千万个物品的 Embedding 存入向量数据库（如 Faiss），**在线**只实时计算 1 个独立于候选集的用户 Embedding，然后去做极速的内积/ANN检索。
* **绝对前提**：**用户的 Embedding 必须是一个固定的静态向量。**

### 1.2 Target Attention 的计算逻辑 (Early Crossing)
* **机制**：以 DIN 模型为例，它拿当前正在评估的 **目标商品 (Target Item)** 作为 Query，去和用户历史序列里的每一个物品算相关性（Target Attention）。
* **灾难性后果**：给用户推荐不同的商品，这个用户就会生成完全不同的 User Embedding（千物千面）。如果在召回阶段使用，面对 1000 万个候选商品，必须实时计算 1000 万次完整的用户向量，这在工业界是绝对无法接受的计算延迟。

---

## 2. 范式对比：Target Attention vs Self-Attention

抛开工程限制，从数学和特征提取的角度来看，两者的本质区别如下：

| 比较维度 | Target Attention (目标注意力) | Self-Attention (自注意力) |
| :--- | :--- | :--- |
| **通俗比喻** | **“拿着考题找复习资料”** | **“自己梳理知识点大纲”** |
| **Query 是谁？** | **对侧塔的信息**（即具体的 Target Item） | **本塔内部的信息**（即历史序列里的其他 Item） |
| **计算目标** | 找出用户历史中，**与当前候选商品最相关**的部分进行激活。 | 找出用户历史序列**内部的关联性、周期性和多重兴趣**。 |
| **User 向量状态** | **动态的 (Dynamic)**：因 Target 的不同而时刻变化。 | **静态的 (Static)**：在某一个特定时刻算出来是固定的。 |
| **适用阶段** | **精排 (Ranking)**：候选集已缩小到百级别，算力足够支撑逐个计算。 | **召回 (Retrieval) / 粗排**：需要提前压缩成固定向量进行海量匹配。 |

---

## 3. 破局之道：为什么召回/双塔用的都是 Self-Attention？

既然 Target Attention 不能用，为了在双塔里挖掘用户复杂的序列信息，工业界（如 **SDM** 和 **ComiRec** 模型）全面拥抱了 Self-Attention：

1. **纯内部消化**：Self-Attention 完全发生在 User Tower 内部。它使用历史序列自己和自己做 Attention（$Q=K=V=\text{用户历史}$）。
2. **不依赖 Target**：它不需要知道接下来要召回什么商品，只负责把用户过去一段混乱的行为，梳理成几个清晰的兴趣方向。
3. **输出固定向量**：计算完成后，它会吐出 1 个（如 SDM）或 $K$ 个（如 ComiRec）**固定的多维向量**。
4. **完美契合 Faiss**：拿着这几个固定好的高质量 User 向量，直接去 Faiss 库里搜即可。既保留了序列的高阶特征，又完全满足了双塔性能极高的工程要求。

### 💡 核心认知总结
* **Target Attention** 是**跨塔交叉（Cross-Tower）**，精度极高，但打破了独立性，只能屈尊于**精排**。
* **Self-Attention** 是**塔内自嗨（In-Tower）**，精度略逊，但保全了独立性，是目前深度**召回**序列建模的绝对天花板。
